Lexicographic Curation Workbench Help

Comprehensive guide to LIFT format and application features

Introduction

Welcome to the Lexicographic Curation Workbench (LCW), a professional tool for creating and managing comprehensive dictionaries using the LIFT (Lexicon Interchange FormaT) standard. This application provides full support for LIFT 0.13 with extensive features designed for lexicographers, linguists, and language documentation specialists.

What is LIFT?

LIFT (Lexicon Interchange FormaT) is an XML-based standard developed by SIL International for storing and exchanging lexical data. It enables:

  • Interoperability: Exchange data between different dictionary tools (FieldWorks, FLEx, WeSay)
  • Data Preservation: Long-term archival using an open, documented standard
  • Flexibility: Support for diverse linguistic structures and custom fields
  • Richness: Capture complex lexical relationships, multimedia, and metadata
Why LIFT Matters for Lexicographers

LIFT is the industry standard for dictionary data exchange. Whether you're documenting an endangered language, creating a bilingual dictionary, or building a comprehensive reference work, LIFT ensures your data remains accessible, portable, and compatible with professional tools used worldwide.

Core Features

Multilingual Support Essential

Every text field in LIFT supports multiple writing systems simultaneously. This is crucial for:

  • Headwords: Enter lexical units in multiple scripts (e.g., Arabic + Latin transliteration)
  • Definitions: Provide definitions in multiple languages (e.g., English + Spanish + French)
  • Examples: Show usage in source language with translations
  • Glosses: Brief translations for quick reference
Example: Multilingual Entry 
Headword: cat [en], chat [fr], gato [es], 猫 [zh]
Definition (English): A small domesticated carnivorous mammal
Definition (Spanish): Un pequeño mamífero carnívoro domesticado

Senses & Subsenses Essential

Organize word meanings hierarchically to capture polysemy and semantic relationships:

  • Multiple Senses: Distinguish different meanings of the same word
  • Subsenses: Create unlimited levels of nested meanings
  • Grammatical Info: Assign part of speech and morphological traits to each sense
  • Semantic Domains: Classify senses by topic or field
  • Sense-Level Relations: Link related meanings (synonyms, antonyms)
Example: Polysemous Word "Bank" 
Entry: bank
  Sense 1 (Noun): Financial institution
    Definition: A place where money is kept and financial services are provided
  Sense 2 (Noun): Riverbank
    Subsense 2.1: Edge of a river
    Subsense 2.2: Slope beside a body of water
  Sense 3 (Verb): To deposit money
    Definition: To place money in a financial institution

AI Assistant Professional

The built-in AI assistant helps you proofread entries and draft new ones using Large Language Models (LLMs). It's designed as BYOK (Bring Your Own Key) — you provide the API key and choose the provider.

AI Proofreading

Click AI Proofread in the entry editor to have the LLM review your entry for:

  • Completeness: Missing definitions, examples, pronunciations
  • Accuracy: Wrong translations, inappropriate part-of-speech
  • Consistency: Style, terminology, formatting
  • LIFT compliance: Valid language codes, correct structure

Results appear in a panel below the form with color-coded issues (red=error, yellow=warning, blue=info). Issues that include a corrected_text have an Apply button that sets the form field to the suggested value — no manual copying needed.

AI Drafting

Click AI Draft to generate a new entry from a description. Describe the word (e.g. "a type of traditional Polish dance" or just "ephemeral") and the LLM produces a complete entry in YAML format that can be applied directly to the form.

Configuration

Configure the AI in Settings → AI / LLM Settings:

  • API Key: Your LLM provider key (stored per project, never shared)
  • API Base URL: The endpoint URL — works with OpenAI, DeepSeek, GitHub Models, NVIDIA NIM, or any OpenAI-compatible provider
  • Model Name: The model ID (e.g. gpt-4o, deepseek-chat, o3-mini)
  • Test Connection: Button to verify your settings before using AI features

Prompt Templates are fully customizable — edit system and user prompts directly in the Settings page. The default prompts are optimized for a bilingual Polish-English dictionary and know about seh-fonipa for IPA, POS inheritance, and optional examples.

Tip: Batch proofreading is available at POST /api/ai/batch-proofread. All AI features work in both interactive (entry editor) and batch (API) modes.

External Scripts & API Integration

The workbench provides a rich REST API for external scripting and integration. Rather than using a plugin system, you can write standalone scripts in any programming language that interact with the dictionary data via HTTP API calls. This approach is more flexible, secure, and empowers users to build custom workflows without modifying the application.

API-First Design

All major functionality is exposed through RESTful endpoints:

  • Entries API: CRUD operations for dictionary entries (/api/entries/*)
  • Query Builder API: Create and execute complex queries (/api/query-builder/*)
  • Worksets API: Manage entry collections (/api/worksets/*)
  • Validation API: Run validation rules and checks (/api/validation/*)
  • Export API: Programmatic access to export formats (/api/export/*)
  • Word Sketch API: Corpus analysis and enrichment (/api/word-sketch/*)

Example: Python Export Script

#!/usr/bin/env python3
"""Example external script to export entries in custom format."""
import requests
import json

# Configuration
BASE_URL = "http://localhost:5000"
API_KEY = "your-api-key"  # If authentication is enabled

# Fetch all entries
response = requests.get(f"{BASE_URL}/api/entries/")
entries = response.json()["entries"]

# Filter and transform
for entry in entries:
    # Custom processing logic
    custom_format = {
        "word": entry["lexical_unit"]["en"],
        "definitions": [s["definition"] for s in entry.get("senses", [])]
    }
    # Write to file, database, or external service
    print(json.dumps(custom_format))

Security Benefits

  • No Server Code Execution: Scripts run in your environment, not on the server
  • Language Agnostic: Use Python, JavaScript, Ruby, curl, or any HTTP client
  • Version Control Friendly: Scripts are plain files you can track in Git
  • Sandboxed: API permissions control what external scripts can access

API Documentation

Interactive API documentation is available via Swagger UI at /apidocs/ when running the application. This provides:

  • Complete endpoint listings with request/response schemas
  • Interactive "Try it out" functionality for testing
  • Authentication requirements per endpoint
  • Example requests and responses

Word Sketches Professional

Word sketches provide collocation analysis powered by ConceptSketch (BlackLab-based Lucene index with 74 million sentences). They show you which words commonly appear with a given lemma and in what grammatical relations.

  • Collocations: Words sorted by logDice association score
  • Grammatical Relations: 41 relation types (modifiers, objects, predicates, etc.)
  • Enrichment: Suggested examples and subentries derived from corpus data
  • Coverage Analysis: Check which entries are missing from the corpus

Access word sketches from the Word Sketch Browser in the Tools menu, or use the /api/word-sketch/sketch/{lemma} API endpoint.

Variants & Allomorphs Advanced

Document different forms of the same lexeme. Our system largely follows the SIL Fieldworks approach to variants and allomorphs:

  • Spelling Variants: Alternative spellings (e.g., color/colour)
  • Dialectal Forms: Regional variations
  • Phonological Variants: Different pronunciations
  • Allomorphs: One of two or more complementary morphs which manifest a morpheme in its different phonological or morphological environments
  • Morphological Allomorphs: Different forms based on context
  • Grammatical Information: POS and traits for each variant

Understanding Allomorphs vs Variants

When deciding between calling something an allomorph or a variant:

  • Allomorphs if it is phonologically conditioned
  • Variants for morphologically conditioned or lexically conditioned suppletion

Tip: When deciding between calling something an allomorph or a variant: Allomorphs if it is phonologically conditioned, otherwise make it a variant for morphologically conditioned or lexically conditioned suppletion. For more information about allomorphs and variants, see the Need More Help? section below.

Example: Allomorph Usage

In the LIFT format, allomorphs can be represented as direct variant forms with traits: 

<variant>
  <form lang="en"><text>grass roots</text></form>
  <trait name="morph-type" value="stem"/>
</variant>

Ranges Editor Professional Open Editor

Manage controlled vocabularies (ranges) for your dictionary. Ranges define the valid values for fields like grammatical categories, semantic domains, and lexical relations.

What are Ranges?

Ranges are LIFT's controlled vocabulary lists that ensure consistency across your dictionary. They define:

  • Grammatical Information: Noun, Verb, Adjective, etc.
  • Semantic Domains: Hierarchical classification systems
  • Lexical Relations: Synonym, Antonym, Part of, etc.
  • Custom Categories: Domain-specific classifications

Managing Ranges

Access the Ranges Editor from Tools → Ranges Editor to:

  • View all ranges: Browse your controlled vocabularies
  • Create new ranges: Add custom classification systems
  • Edit range details: Update labels and descriptions in multiple languages
  • Manage elements: Add, edit, or delete individual values within ranges
  • Check usage: See which entries use specific range values
  • Safe deletion: Migrate values before deleting to maintain data integrity

Hierarchical Ranges

The Ranges Editor supports two hierarchy patterns:

  • Nested Hierarchy: Elements directly nested within parents (e.g., Noun → Common Noun, Proper Noun)
  • Parent-Based Hierarchy: Flat structure with parent attributes (e.g., semantic domains)

Usage Analysis

Before deleting range elements, the editor shows:

  • Usage count: How many entries use each element
  • Sample entries: Examples of entries using the element
  • Unused elements: Elements safe to delete without affecting data
Example: Managing Grammatical Categories

If you want to split "Noun" into "Common Noun" and "Proper Noun":

  1. Open Tools → Ranges Editor
  2. Select the grammatical-info range
  3. Click Elements tab to see all grammatical categories
  4. Click Usage tab to see which categories are actually used
  5. Add new elements "Common Noun" and "Proper Noun"
  6. Update entries to use the new, more specific categories
  7. When ready, delete "Noun" and choose to migrate remaining references

Value Migration

When deleting a range element that's in use, you can:

  • Replace: Migrate all references to another element
  • Remove: Delete the value from all entries
  • Cancel: Keep the element if you change your mind

Validation Rules Advanced

Ensure dictionary quality with configurable validation rules that check entries for completeness, consistency, and adherence to project standards. The system includes a built-in rule editor and batch validation capabilities.

What Are Validation Rules?

Validation rules are automated checks that evaluate dictionary entries against defined criteria. Each rule specifies:

  • Scope: What part of the entry to check (e.g., lexical unit, senses, pronunciation)
  • Condition: When the rule applies (e.g., field is present, field is empty, value matches pattern)
  • Validation Type: How to check (e.g., required, regex pattern, length check, custom function)
  • Priority: Error severity (critical, warning, info)
  • Message: Human-readable feedback when validation fails

Rule Editor

Access the Rule Editor from Tools → Rule Editor to:

  • View all rules: Browse active validation rules for your project
  • Create new rules: Add custom validation checks tailored to your project
  • Edit existing rules: Modify rule parameters, conditions, or error messages
  • Enable/disable rules: Toggle rules on/off without deleting them
  • Test rules: Preview rule behavior against sample entry data
  • Export/Import: Share rule sets between projects
Example: Required IPA Rule

A rule to ensure all entries have IPA pronunciation:

  • Path: $.pronunciations[*].ipa
  • Condition: if_present (check when pronunciation exists)
  • Validation: required (IPA must be non-empty)
  • Priority: warning
  • Message: "Pronunciation entry must have an IPA form"

Batch Validation

Run validation across your entire dictionary using Tools → Batch Validation:

  • Select scope: Validate all entries or filter by search criteria
  • Filter by severity: Show only critical errors, warnings, or all issues
  • Export results: Download validation report for quality assurance records
  • Drill down: Click any issue to open the problematic entry for editing

IPA Pronunciation Anomaly Detection Professional

Rule R4.3.1 uses a trained grapheme-to-phoneme model to flag IPA pronunciations that diverge sharply from the pronunciation predicted for the headword. Parenthetical IPA (e.g. ˈskɒtɪˌsɪz(ə)m) is decompressed automatically before comparison. The rule is a warning, so it assists editors without blocking saves.

The model is not shipped with the application. An administrator trains it on the project's own dictionary (see scripts/ipa_training/) and places the resulting ipa_anomaly_*.pt + .json bundle in the server's instance/ipa_models/ directory. Users cannot choose the model or its location — discovery is entirely server-side and safe.

Custom Validation Functions

The Validation Rules Editor includes a comprehensive library of pre-built custom validation functions for complex validation scenarios. These functions handle specialized checks beyond basic type and pattern validation.

Using Custom Functions (No Code Required)

When creating or editing a validation rule, select "Custom Function" as the validation type, then choose from the dropdown of available functions:

  • Content Validation: Validate Sense Content or Variant, Validate Note Content
  • Structural Checks: Validate Subsense Depth, Validate Multilingual Note Structure
  • Relationship Integrity: No Circular Component References, No Circular Sense/Entry Relations, Validate Relation Targets Exist
  • Consistency Rules: POS Consistency, Conflicting POS, Synonym/Antonym Exclusion
  • Data Quality: Validate Date Fields, Validate Language Code Format, Validate Unique Note Types
  • Linguistic Validation: IPA Characters, No Double Stress/Length markers, Hunspell Spelling
Example: Using Custom Functions

To validate that entry relations don't create circular references:

  1. Create a new rule with category "relations_level"
  2. Set validation type to "Custom Function"
  3. Select "No Circular Entry Relations" from the dropdown
  4. Test the rule against sample entries before saving

Adding New Custom Functions (Developers Only)

To add entirely new custom function types, Python code development is required:

  1. Implement the function in app/services/validation_engine.py
  2. Register it in the custom functions registry
  3. Restart the application to load the new function
  4. The function will appear in the dropdown for all users

Note: Adding new function types requires server access and code deployment. End users can configure rules using all existing custom functions without writing code.

Rule Management Best Practices

  • Start simple: Begin with basic required-field rules, add complexity gradually
  • Use priorities wisely: Critical for blocking errors, warnings for suggestions
  • Document rules: Use descriptive names and messages that help editors understand issues
  • Test before deployment: Use the Rule Editor's test feature to preview rule behavior
  • Version control: Export rule sets when making significant changes for backup

Workbench Professional

The Workbench provides advanced tools for managing large-scale dictionary work through query-based workflows, worksets, and batch operations.

Query Builder

Build sophisticated queries to filter and select dictionary entries based on multiple criteria. The Query Builder supports complex conditions, cross-field references, and performance indicators to help you work efficiently with large datasets.

  • Filter Conditions: Multiple criteria across entry fields, senses, examples, relations, and etymology
  • Operators: Basic string matching (equals, contains, starts/ends with), regular expression (regex), numerical comparisons, list operations, and fuzzy matching
  • Cross-Entry Operations: Find similar headwords, duplicates, components, and phonetic matches
  • Element References: Reference values from other filter conditions for complex logic
  • Regex Search: Use XPath regex patterns for field-level search — e.g., ^un.* finds headwords starting with un-, .*ing$ finds glosses ending in -ing. Supports anchors (^, $), character classes ([aeiou], \d, \w), quantifiers (*, +, ?, {n,m}), alternation (foo|bar), and the i flag (case-insensitive, always on). Note: XPath regex differs from Python/PCRE — no lookahead/lookbehind, no \b word boundary. Supported on lexical unit, definition, gloss, example, citation, and note fields.
  • Performance Indicators: Real-time feedback on query complexity (fast/medium/slow)
  • Query Validation: Preview result counts and validate query syntax before execution
  • Saved Queries: Store and reuse complex filter combinations
  • Sorting: Order results by lexical unit, part of speech, creation date, or modification date

Access: Workbench → New Workset or Workbench → Query Builder

Example Queries
  • Missing Definitions: Find entries where sense.definition is empty
  • No Pronunciation: Entries lacking IPA or pronunciation data
  • Similar Words: Find potential duplicates using fuzzy matching
  • Compound Candidates: Headwords that appear as components in other entries

Duplicate Detection & Data Quality Professional

The Data Quality Dashboard offers advanced tools to analyze dictionary health, identify redundant data, and deduplicate headwords and example sentences.

  • Exact & Near Duplicate Detection: Scan headwords using advanced normalization:
    • Parenthetical Normalization: Normalizes headwords with parentheses (e.g., matching "a little (bit)" with "little" or "little bit").
    • Compound Normalization: Normalizes compounds by ignoring spaces and hyphens (e.g., matching "log-in" with "log in").
    • POS Wildcard Matching: Entries with no part of speech can group with matching headwords of any POS.
    • Homograph Filtering: Filters candidate groups by definition similarity using character-trigram Jaccard scores to separate homographs from true duplicate entries.
  • Redundant Examples Finder: Identifies identical or highly similar example sentences and phrase subentries across the dictionary (using Jaro-Winkler similarity with a 95% threshold) to clean up repetitive translations.
  • One-Click Curation: Dynamic grouping view that allows you to compare entries side-by-side, suggest which duplicate to keep (e.g. prioritizing metadata completeness), dismiss false positives, or merge entries directly.

Access: Tools → Data Quality or Dashboard → Data Quality

Importing Real Abbreviations via list.xml

When you export a LIFT file from FieldWorks Language Explorer (FLEx), the Variant Types, Complex Form Types, Parts of Speech, and other controlled vocabularies are exported with their full names only — not their abbreviations. This means the Range Editor and dictionary preview will show placeholder values (e.g., full names truncated to 3 characters) instead of the real abbreviations linguists have curated in FieldWorks.

How to Get Real Abbreviations
  1. In FLEx, go to File → Export → List of Lists.
  2. Save the output as list.xml.
  3. On the Import page, upload list.xml alongside your LIFT file (and optional .lift-ranges file).
What Happens During Import
  • The parser maps FieldWorks list names (e.g., "Variant Type Names") to internal range IDs.
  • Multilingual labels, abbreviations, and descriptions are extracted from the XML.
  • FieldWorks 3-letter language codes (Eng, Pol, Frn) are normalized to 2-letter ISO 639-1 codes (en, pl, fr).
  • The data is stored as LIFT ranges XML in the database, merged with any existing .lift-ranges data.
  • The Range Editor immediately shows real abbreviations like "irreg. infl.", "comp.", "contr.", "der.", "ph. v."
CLI Alternative

You can also import from the command line: 

flask import-list-xml /path/to/list.xml

Importing SFM / Shoebox and CSV Data

The workbench supports importing dictionary data from Standard Format Marker (SFM) files (also called Shoebox files) and CSV spreadsheets. This allows you to migrate legacy dictionary data from tools like Shoebox, Toolbox, or spreadsheet-based workflows.

How SFM Import Works
  1. Go to Tools → Import Shoebox / SFM from the navigation menu.
  2. Upload your .sfm or .txt file.
  3. The system parses the file and displays all detected markers with sample values in an interactive mapping table.
  4. For each marker, choose:
    • LIFT Element — what kind of data the marker represents (e.g., \lx → headword, \ph → pronunciation, \ps → part of speech, \de → definition). The dropdown exposes all 26 LIFT data-carrying elements.
    • Level — whether the field belongs at the entry, sense, example, pronunciation, or variant level.
    • Key — check this for markers that start a new entry or sense (e.g., \lx starts a new entry).
    • Language — the language code for multilingual data (e.g., \d_Pol → pl).
    • Field Type — normal data, cross-reference source/target, or variant marker.
  5. Click Execute Import. The system converts the SFM data to LIFT XML and merges it into the dictionary database.
CSV Import

CSV import follows the same two-step workflow (upload → map columns → import), with one row per sense. Define which column holds the headword, part of speech, definition, etc., just like with SFM markers.

Inline Allomorph / Variant Support

SFM files can include inline allomorphs using \am and \al markers. These create <variant> elements in the generated LIFT. Map \am to variant_form at the variant level. Cross-reference variants via \mn (variant target) and \vt (variant type) field types.

Cross-References

Use \lf (source) and \lv (target) markers with field types cross-ref-source and cross-ref-target. References are resolved in Phase 2 after entries are imported; unresolved references are stored as <annotation type="import-residue"> elements.

Tips
  • Language codes in marker names (e.g., \ps_Eng) are automatically extracted — you don't need to set the language manually for those.
  • Preview the entry table before importing to verify your mapping produces the expected structure.
  • Save mappings for reuse by giving them a name before importing.

Exporting to Markdown

The workbench can export your dictionary as a Markdown document suitable for publishing, documentation, or further processing with tools like Pandoc.

How to Export
  1. Go to Tools → Export → Markdown from the navigation menu.
  2. Optionally select a display profile to control which fields appear and how they are formatted.
  3. Optionally filter by entry range (e.g., A–M) or part of speech.
  4. Click Export. The system generates a Markdown file that includes headwords, pronunciations, parts of speech, definitions, glosses, examples, and other mapped fields.
  5. Download the resulting .md file. You can convert it to PDF, HTML, or other formats using Pandoc: 
    pandoc dictionary.md -o dictionary.pdf --pdf-engine=xelatex
Features
  • Multilingual content preserved with language annotations.
  • Pronunciation data (IPA) included where available.
  • Example sentences with translations.
  • Headwords grouped and sorted alphabetically.
  • Works with Pandoc, Jekyll, Hugo, and other Markdown-based publishing tools.
